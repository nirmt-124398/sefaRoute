# Database Migrations (Alembic)

This project uses **Alembic** with async SQLAlchemy (`asyncpg`) for database schema migrations.

## How it works

On every app start (`main.py` lifespan), `alembic upgrade head` runs automatically via a subprocess. This ensures the database schema is always up to date with the migration scripts.

## Commands

```bash
# Create a new migration from model changes
uv run alembic revision --autogenerate -m "description of change"

# Review the generated migration before applying
uv run alembic upgrade head --sql

# Apply pending migrations
uv run alembic upgrade head

# Roll back one step
uv run alembic downgrade -1

# Check current migration state
uv run alembic current

# View migration history
uv run alembic history
```

## Workflow for schema changes

1. Edit the SQLAlchemy model in `db/models.py`
2. Generate a migration:
   ```bash
   uv run alembic revision --autogenerate -m "add role column to users"
   ```
3. Review the generated file in `alembic/versions/` — verify the `upgrade()` and `downgrade()` functions are correct
4. Apply it:
   ```bash
   uv run alembic upgrade head
   ```
5. Restart the app — the lifespan will run `upgrade head` automatically

## Migration files

All migration scripts live in `alembic/versions/`. These should be committed to git alongside model changes. Each migration must have both `upgrade()` and `downgrade()` functions.

- **Up**: applies the change
- **Down**: reverses the change (rollback)

## Initial migration

The initial migration (`3c35c834e2cd`) was manually created after Alembic was introduced. It creates all three tables (`users`, `virtual_keys`, `request_logs`) with their indexes and foreign keys. On an existing database, `alembic stamp head` was used to mark the current state without re-running the CREATE TABLE statements.

## Config files

| File | Purpose |
|------|---------|
| `alembic.ini` | Alembic configuration (URL sourced from `.env`) |
| `alembic/env.py` | Async engine setup, model imports, DB_SCHEMA handling |
| `alembic/script.py.mako` | Template for new migration files |

## Notes

- Alembic uses the `DATABASE_URL` from `.env` (same as the app)
- `DB_SCHEMA` env var is respected — migrations run inside the specified schema
- The migration runs in a subprocess on startup to avoid conflicts with FastAPI's async event loop
- Always review autogenerated migrations before applying — they may miss complex changes like renaming columns
